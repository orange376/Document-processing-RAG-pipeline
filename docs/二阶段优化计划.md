# RAG Pipeline — 二阶段优化计划

> 基于全量代码审查（2026-07），汇总当前实现与"工业级 RAG 文档处理流水线"目标之间的全部缺口，分两阶段逐步攻破。

---

## 阶段一：核心功能补全（P0）

> **目标**：让流水线在真实 Word/PDF/扫描件三大场景下能跑通、跑对，核心指标不再"虚"。

| # | 缺口 | 影响 | 涉及文件 | 方案要点 |
|---|------|------|----------|----------|
| 1 | **Word 文档版面分析永远为空** | `layout_quality` = 0 → 总置信度被系统性拉低；Word 没有版面树 → 切片无结构信息 | `orchestrator.py:_page_to_array()` | 对 docx 先用 `python-docx` 获取页面尺寸、再通过 `win32com` 或 `libreoffice` 转 PDF → 用 fitz 渲染；或先用 `pillow` 合成空白画布 + 将 word_loader 提取的 bbox 信息映射上去 |
| 2 | **扫描件 PDF 未调用多模态大模型** | 纯扫描件走 easyocr → 文本识别率低、版面信息完全丢失 | `orchestrator.py:process_document()` | 在阶段 2 中探测 page.images → 对无文本 + 含大图的页面调用 Qwen-VL（page-level recognition），prompt 要求输出 markdown 结构 |
| 3 | **OMML→LaTeX 转换缺失** | Word 内嵌公式在解析后仍是乱码 XML 片段，无法检索 | `word_loader.py:_collect_text_from_element()` | 解析 `m:oMath` 子树 → 递归映射 OMML 节点到 LaTeX 符号；使用预定义映射表 + fallback 到 Qwen-VL 图片识别 |
| 4 | **BM25 索引不持久化** | 服务重启后 BM25 丢失 → 纯向量检索 → 关键词召回为 0 | `bm25_index.py` | 新增 `save()` 调用时机（index_chunks / add_documents）；加载时从 `task_db.json` 同级目录 deserialize；添加版本校验 |
| 5 | **`needs_review` 绕过阈值系统** | 置信度 0.7 但仍被放行，或因为 `chunk_coherence < 0.8` 被误拦截 | `upload.py:needs_review` | 删除硬编码阈值，改用 `ThresholdStrategy.classify(overall_confidence)`；拒绝列表需人工可见 |
| 6 | **OP** — `_page_to_array()` 方法统一重构 | 为 #1、#2、#3 提供干净的图片渲染入口，当前 try/except 结构不清晰 | `orchestrator.py:307-335` | 拆分为 `_render_pdf_page()` 和 `_render_docx_page()`；Word 转 PDF 逻辑复用；缓存渲染结果避免重复 |

## 阶段二：工业级能力增强（P1 + P2）

> **目标**：让流水线在精度、鲁棒性、可观测性、可维护性上达到生产标准。

### P1 — 用户体验与可靠性

| # | 缺口 | 影响 | 涉及文件 | 方案要点 |
|---|------|------|----------|----------|
| 7 | **上下文窗口无管理** | 检索结果过多 → LLM 输入超限截断；结果太少 → 回答不完整 | `context_builder.py` | 加入 token 计数（tiktoken）；按 reranker 得分排序 + 累计截断；配置 `max_context_tokens` 参数 |
| 8 | **Hybrid Search 缺少 `document_ids` 过滤** | 多文档场景无法限定搜索范围 → 回答串文 | `hybrid_search.py` | vector store 和 BM25 都支持 payload filter；在 `Retriever` 层暴露 `source_files` 参数 |
| 9 | **表格检测/恢复是桩代码** | 表格抽取后只有行列数，没有单元格内容和合并关系 | `table/detector.py`, `table/structure.py` | 集成 PaddleX 表格结构识别模型（SLANet）；或使用 Qwen-VL 对表格区域截图进行结构识别；按 PP-DocLayoutV3 的 table 框裁剪 |
| 10 | **图片未按类型路由** | 公式图跑 OCR、表格图走公式识别——类型错配，识别率低 | `orchestrator.py:_ocr_embedded_images()` | 根据 LayoutElement.category（formula/table/figure）路由到对应处理器；无分类时先用 Qwen-VL 分类再路由 |
| 11 | **异步索引阻塞** | Embedding + 索引阶段占用主循环 → 无法响应其他请求 | `vector_store.py:index_chunks()`, `orchestrator.py` | Qdrant client 支持异步 gRPC；或提交到后台 asyncio task，轮询完成状态 |
| 12 | **切片的 `layout_tree_path` 永远为 None** | ChunkMetadata 字段存在但不赋值 → 上下文展示缺少层级信息 | `chunker.py:_make_chunk()` | chunker 获取当前 page 的 layout_tree 交叉引用；在 _make_chunk 时传入 path 字符串 |
| 13 | **切片无重叠 / 仅按长度硬截断** | 边界处语义断裂 → 检索时关键句落在切片间隙中 | `chunker.py` | 添加 overlap（128 chars）；支持按句号/换行符对齐边界；对长块用滑动窗口再分 |
| 14 | **QueryRewriter 形同虚设** | `query.py` 构造 `QueryRewriter()` 时未传入 LLM → 重写永远返回原句 | `query.py:116`, `query_rewriter.py` | 传入 `llm_client` 实例；prompt 要求输出同义改写 + 扩写；将原始查询与改写结果同时送入 hybrid search |

### P2 — 基建与运维

| # | 缺口 | 影响 | 涉及文件 | 方案要点 |
|---|------|------|----------|----------|
| 15 | **结构化日志与监控缺失** | 无法追踪错误、排查性能瓶颈 | 全局 | 集成 loguru 或 structlog；添加请求耗时埋点；关键步骤（OCR、layout、embedding）记录耗时和成功率 |
| 16 | **人工审核的反馈回路** | 审核修改（编辑 block、标记识别错误）未被记录和利用 | `review.py` | 将 edited_blocks 存入独立存储；定期生成错误统计面板；可用于增量微调 OCR/公式模型 |
| 17 | **多粒度切片（Sliding Window）** | 当前只有一种切片粒度 → 语义完备性和检索精度之间无平衡 | `chunker.py` | 支持三步切片：文档级→段落级→句子级；检索时逐步缩小范围 |
| 18 | **Docker 化部署** | 环境依赖繁杂（paddle、torch、easyocr、qdrant），新人上手门槛高 | 新增 `Dockerfile`, `docker-compose.yml` | 分层构建：base→deps→model→app；GPU support；数据卷挂载 |
| 19 | **自动化测试** | 核心模块（chunker、hybrid_search、scorer）0 测试 → 修改无安全网 | 新增 `tests/` | pytest + 模拟 document/chunk 数据；按模块组织：`test_chunker.py`, `test_hybrid.py`, `test_scorer.py` |
| 20 | **认证与频率限制** | API 完全开放 → 生产环境风险 | `api/` | FastAPI 的 `Depends()` 注入 token 验证；`slowapi` 加频率限制 |
| 21 | **Redis 缓存闲置** | `settings.py` 配置了 `redis_url` 但从未使用 | 全局 | 缓存 embedding 向量、reranker 结果、LLM 回答；TTL 按场景设置 |

## 执行原则

1. **阶段一优先**：在 P0 全部完成前，不启动 P1 任务。#6（`_page_to_array` 重构）是 #1 和 #2 的前置依赖，需最先落地。
2. **每项完成标准**：
   - 代码合入（通过现有回归）
   - 对应场景的测试文档或单元测试更新
   - 在本地 8001 端口验证通过
3. **排期建议**：
   - 阶段一 ≈ 5-8 个工作日（一次性落地）
   - 阶段二 P1 ≈ 10-15 个工作日（可分项并行推进）
   - 阶段二 P2 ≈ 持续演进（按需优先）
